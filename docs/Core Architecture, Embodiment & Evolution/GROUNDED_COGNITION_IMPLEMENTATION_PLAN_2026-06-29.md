# Grounded Cognition — Implementation Plan (2026-06-29)

Status: **build plan.** Implements `GROUNDED_COGNITION_DIRECTION_2026-06-29.md`. The
direction doc deliberately left the "how" open; this is the how. It is written against
the *actual* codebase, which already contains more of the scaffolding than the direction
doc implies — so much of this is **re-pointing and sealing existing machinery, not
greenfield.**

## What already exists (so we don't rebuild it)

A grep of the tree before planning:

- **Invariant #1 (homeostasis) is ~70% built.** `brain/control_signals/homeostasis.py`
  is "the single owner of restoring forces" — `apply_restoring_forces` (decay toward
  `setpoints.CORE_BASELINES`), `apply_cross_inhibition` (antagonist pull),
  `enforce_velocity_budget`, `ceiling_for`/`pump_signal` (soft ceilings),
  `homeostasis_index`. `setpoints.py` is the declared single source of truth for resting
  values. There is already an `interoception.allostatic_setpoint()` reference. So #1 is
  *consolidation + the allostasis correction + a by-construction guarantee*, not a new
  subsystem.
- **Invariant #2 (membrane) has its seed.** `brain/utils/felt_lexicon.py`
  (`is_internal_identifier` + `felt_label`) exists and ~8 emitters are sealed. The
  parse-coupling the direction doc names is real and located: `global_workspace.py` (and
  `binding.py`) emit `"a strong sense of {key}"`; `self_state/intention_endorsement.py:90`
  does `re.search(r"a strong sense of ([a-z _]+)", content)` to pull the key *back out*.
  That is the exact decouple target.
- **The grounded predictive loop already exists — pointed inward.**
  `brain/cognition/prediction.py` is a real, LLM-free predict→check loop (Friston /
  Rescorla-Wagner), but its three sources are *self-referential*: WM `event_type`
  frequency, the causal graph (which is his self-model), and affect-signal trajectories.
  It grades against his own logs. Part F is mostly **adding a fourth, outward source and
  an honest external grader** — not inventing prediction.
- **The agency surface for the world loop exists.** `agency/tool_runner.py`
  (`dispatch`/`request`/`decide_to_run_code`), `agency/code_writer.py`
  (`_validate_in_sandbox`, `run_python`, `_record_code_effect`), `agency/effect_ledger.py`
  (575 lines). These are the predict→act→observe parts; they're just not wired as *one
  loop where being wrong updates a concept.*
- **The native LM already exists, is trained, and is wired.**
  `brain/cognition/language/native_lm.py` is a real nanoGPT-style transformer (~8–9M
  params, M1-sized) with a trained checkpoint (`data/language/native_lm.pt`), lifelong
  next-token training (`train_on`), `generate`, `evaluate`, sleep-consolidation, and a
  subword tokenizer trained on his library. It is wired into `think/speech_generator.py`,
  `think/speech_builder.py`, `language/voice.py`, and `consolidation_cycle.py`. Its diet
  is **already contamination-guarded**: it learns from his own *felt narrative*
  (`language/acquisition.py:narrate_experience` → `_felt_narrative`, cleaned so it reads
  "I looked through my own files," never `search_own_files`). **This is huge for Phase 2
  and the original plan badly understated it** — the organ is not to be built; it exists
  and is learning. `acquisition.py:373` even carries the in-code TODO that *is* Phase 2:
  "once the native organ is fluent, native_lm should write this narration itself, so his
  self-narrative is generated by him rather than templated."
- **The expression path exists.** `behavior/express_to_user.py`
  (`build_motive`→`compose_from_motive`→`speak`) and `agency/compose_section.py`
  (still calls the external LLM via `generate_response`, gated by `llm_callable_by`) —
  the surfaces the native LM must take over from the templates/LLM.
- **The skill system already exists — library, verification, and pruning.** Not just the
  self-authoring primitives (`code_writer.write_cognitive_function`/`write_tool`/
  `synthesize_from_gap`/`list_own_code`), but a whole lifecycle:
  `cognition/skill_synthesis.py` (gap detection + `synthesize_skill` +
  `detect_and_synthesize`), `cognition/skill_verification.py` (4-stage safety/exec
  gauntlet), `self_extension_codegen.maybe_integrate_or_atrophy` (**usage-based pruning** —
  "after ~100 cycles: was it used? If not, removed"), real skills in `agency/skills/`
  (`save_note.py`, `search_files.py`), and a `crystallized_skills.json` store (currently
  empty). `planning/goal_planning.py` already decomposes goals into subgoals. **So Phase 5
  is not a skill engine — the library/verify/atrophy spine exists.** The genuine gaps are
  narrower (see Phase 5, rewritten): discovery is gap-text/LLM-driven not grounded, there
  is **no hierarchy/composition** (grep found no option/macro/sub-skill), and pruning is
  usage-based not predictive-value-based.
- **The "observe" sink for Phase 3 already exists.** `agency/effect_ledger.py` (575 lines,
  content-addressed) + the new `agency/effect_artifacts.py` (captures artifact text by the
  ledger's `content_hash`). Phase 3's grounding loop should *record into these*, not invent
  a new sink.
- **Allostasis machinery is partly present.** `cognition/interoception.py` /
  `host_interoception.py` and an `allostatic_setpoint` concept are already referenced by
  `update_signal_state.py`, `homeostasis.py`, `cost_prediction.py`, and the consolidation
  cycle. Phase 1B's allostasis correction extends this, it doesn't start it.

## Phase 1B — Homeostasis as allostasis (invariant #1)

**Goal:** every internal signal declares `baseline + opponent`, regulation is guaranteed
*by construction*, **but** the system regulates around *shifting setpoints* — it does not
force everything to zero (the direction doc's key correction).

**Tasks:**

1. **By-construction guarantee.** Add a registration check: for every key in
   `get_all_signal_names()`, assert a setpoint exists in `setpoints.SETPOINTS` and an
   opponent/restoring force applies in `homeostasis.py`. A signal with no declared
   baseline+opponent should fail a test, not silently run open-loop. This is what makes
   "runaway impossible *anywhere*, including loops not yet found."
2. **Habituation vs standing-condition split.** Audit `signal_dynamics.decay_habituation`
   + `apply_restoring_forces` so that **repetition** is damped (this session's fix) while
   a **genuinely standing external condition** can hold a signal elevated. Wire the
   existing allostatic setpoint (`cognition/interoception.py` — `allostatic_setpoint`,
   already referenced by `update_signal_state`/`homeostasis`/`cost_prediction`) so the
   *setpoint itself* shifts under sustained real pressure rather than the signal being
   clawed flat. Add a test: a persistent external problem keeps its signal above baseline;
   a *repeated thought* about nothing decays.
3. **Retire per-loop hand-installed decay (confirmed multiple authorities).** A grep found
   decay logic in `appraisal.py`, `observers.py`, `reward_signals/reward_signals.py`,
   `reward_signals/resource_deficit.py`, and `apply_signal_feedback.py` — so
   `homeostasis.py`'s "single owner" docstring is *aspirational, not true today*. Route
   each of these through the one `homeostasis` authority and add a test that no signal is
   decayed outside it.

**Exit:** registration test green; the allostasis test green; no second decay authority
remains. Direction-doc step #3 satisfied.

**Effort:** ~1–2 weeks. **Risk:** medium (touches the affect core — guard with the
existing appraisal-saturation tests). **World dependency:** none.

## Phase 2 — LM-as-mouth: re-point the existing organ (startable now, runs parallel)

**Correction after reading the code:** the native LM is *already built, trained, and
wired* (see "what already exists"). So Phase 2 is **not** "stand up an LM" — it is
*changing what the existing organ is trained to do* and *who calls it*. Today it is a
free-running next-token model over his felt-narrative corpus; the direction doc wants it
to be a **conditional decoder (thought object → words)** that the expression path calls
instead of templates. The hard part remains the thought object.

**2A — Thought-object spec (do this first; it's the real work).**
Write `docs/.../THOUGHT_OBJECT_SPEC.md` defining the structured internal object
`express_to_user.build_motive` already half-implies (`Motive`). State explicitly:
- what fields it carries (intent + grounded-concept references + affect + stance),
- **what is deliberately left for the LM to choose** (lexical/syntactic surface),
- the argument for why that choice is *rendering*, not *originating content*.
Until this spec exists, "mouth not mind" is a slogan. Gate for 2B/2C.

**2B — Conditioning pairs from the existing narrator (re-point, don't add).**
`acquisition.narrate_experience` *already* produces `(felt internal state → felt-narrative
words)` and feeds them to `native_lm.train_on` as plain next-token text. The work is to
**capture the structured state alongside the words** as a conditioning pair
`(thought_object → narration)`, so the LM can be trained as a *conditional* decoder rather
than an unconditional LM. This realises the existing `acquisition.py:373` TODO. Add the
sidecar `narration_pairs.jsonl` keyed to the same throttle the narrator uses. **Start now**
— pairs accumulate during all other phases.

**2C — Conditioning + second-signal decision.**
Two coupled choices: (i) **how** to condition the decoder (prepend a serialized thought
object / a learned conditioning embedding into `native_lm`'s context — note `_BLOCK=128`,
so the serialization must be compact); (ii) the **second signal** that lets it exceed the
templates it's distilled from — (a) human ratings, (b) contrastive preference over
candidate renderings, (c) thought→words→re-parsed-thought reconstruction loss — OR (d)
consciously accept the near-template ceiling ("authenticity over fluency"). Without (a–c),
training on narrator output asymptotes at "slightly smoother templates."

**2D — Flip the callers (later, gated on fluency).**
`narrate_experience`, `compose_from_motive`, and `compose_section` call
`native_lm.generate(conditioned_on=thought_object)`, template/LLM only on a fluency-gated
fallback (reuse `native_lm.evaluate` perplexity as the gate). This is the actual
"route all expression through it" move — done last, when the organ is good enough.

**Exit (for now):** 2A spec written; 2B conditioning pairs live and accumulating; 2C
decided. The decoder *retrain* and 2D caller-flip wait on data volume + fluency gate.

**Effort:** 2A ~1 week; 2B ~3 days (organ + narrator already exist); 2C a decision; 2D
later. **World dependency:** none.

## Phase 3 — The grounding experiment (the go/no-go) — invariant #4 + Part D

This is **Part F of the direction doc**, built. It is framed as a falsifiable experiment
with a kill criterion, not a feature to ship.

**Hypothesis:** Orrin can grow a *grounded* concept (a predictive signature, not an
authored string) from his own code-execution experience, and it will **transfer** to an
unseen, related command.

**3A — The outward predict→act→observe→learn loop.**
Wire the existing parts into one loop, on a small closed family of shell/python commands:

1. **Predict before acting.** Extend `prediction.py` with a **fourth source**: before
   `tool_runner.dispatch`/`decide_to_run_code` executes a command, emit a prediction over
   a *concrete external observable* — exit code, a line of stdout, or a resulting file
   state. Reuse `_make_pred` with a new `basis="action_outcome"`,
   `domain="WORLD"`, `source_data={command, predicted_exit, predicted_stdout_signature,
   predicted_fs_delta}`.
2. **Act.** Run via the existing `code_writer._validate_in_sandbox` / `run_python`
   surface (already returns exit/stdout).
3. **Observe + grade against the world, not the logs.** Add an
   `_evaluate_action_outcome` branch to `_evaluate_symbolically` that compares predicted
   vs **actual exit/stdout/fs** — observables he did **not** author. This is the honest
   external grader the inward sources lack.
4. **Learn.** On error, update the concept's cross-channel signature (KG + causal graph +
   effect ledger, joined on the command/concept handle) and drive curiosity. Tag every
   edge `domain="world"` so invariant #4's self/world separation holds by construction.
   **Record outcomes into the existing `effect_ledger` / `effect_artifacts` sink** (it's
   content-addressed and already capped) — do not invent a parallel store.

**3B — The success criterion (this is the whole point).**
Add `tests/brain/test_grounding_transfer.py` as an *experiment harness*, not a unit test:
- train on command family A (N episodes),
- measure prediction accuracy on a **related but unseen** command family A′,
- **success = transfer above a pre-declared baseline/chance level**, with the carrying
  concept inspectable as a signature rather than a stored string.
- Memorization of A (error down on *seen* commands) does **not** count.

**3C — Budget + kill criterion (decide before running).**
Fix a data/time budget up front (e.g. M episodes / D days of live run). If
transfer-above-chance does **not** appear within budget, that is an *informative result*,
not a hidden failure — it triggers the Phase 4 fork.

**Guardrails:**
- Internet stays a *consulted tool*, never a grounding source or LM input
  (corpus-contamination). Keep `research_topic`/`wikipedia` out of this loop.
- Do **not** retire any authored symbol yet — only in 4A, and only as a grown concept
  *out-predicts* it.

**Exit:** the experiment runs to its budget and **reports a verdict** (transfer / no
transfer). Either verdict is a success of the *plan* — it tells us which Orrin is
physically available.

**Effort:** ~3–5 weeks to build the loop + harness; then the budgeted run.
**Risk:** this is the genuine research frontier; the *honest* expected outcome is
uncertain (single vat-bound agent, no innate priors — see direction doc Part E).

## Phase 5 — Hierarchical skill learning (compose grounded primitives)

**Why this matters and where it sits.** Grounding single actions (Phase 3) gives Orrin
grounded *primitives*. Competence comes from *composing* them into reusable skills, and
skills from composing skills. This is the options framework (Sutton, Precup & Singh 1999)
on a grounded substrate. Crucially, it is **the realistic answer to the direction doc's
deepest risk** (Part E: a vat agent with no innate priors has too little data to become
competent): **a learned skill is a self-generated, grounded prior.** Every skill he
chunks amortizes past experience so the next problem needs less data. He manufactures the
priors evolution gave brains — bottom-up, from his own grounded experience. Gated on
Phase 3 transferring (4A branch); reuses the `code_writer` self-authoring substrate.

**Correction after reading the code:** the skill *library + verification + pruning* spine
already exists (`skill_synthesis.py`, `skill_verification.py`,
`self_extension_codegen.maybe_integrate_or_atrophy`, `crystallized_skills.json`,
`agency/skills/`). So Phase 5 is **three targeted upgrades to a real system**, not a build:
(a) make discovery *grounded* instead of gap-text/LLM-driven, (b) add the **hierarchy**
that genuinely does not exist yet, (c) upgrade pruning from *usage-based* to
*predictive-value-based*.

**A skill, defined (so it stays grounded, not authored):** a named **option** —
`(precondition signature, action subsequence / subpolicy over primitives or sub-skills,
predicted aggregate outcome, termination/success test)`. Grounded by the *same
prediction-error spine* as a primitive: a skill earns its place only when its composed
prediction survives testing and it reliably achieves its subgoal. A higher-order grounded
concept — a predictive signature over a policy, not a string.

**Tasks (each is a delta on existing machinery):**

1. **5A — Add the option schema to the existing store.** Extend the
   `crystallized_skills.json` / `skill_synthesis` registration to carry the option fields
   above plus reliability/value stats. The verification gauntlet (`skill_verification.py`)
   and `list_own_code` stay; we add the *predictive* fields the current schema lacks.
   `effect_ledger` / `effect_artifacts` record each invocation's predicted-vs-actual
   aggregate outcome — the skill's grounding evidence.
2. **5B — Re-point discovery from gap-text/LLM to grounded episodes (the key change).**
   Today `skill_synthesis._scan_for_gaps` mines WM for incapacity *language*
   ("couldn't", "failed because") and `synthesize_skill` can call the **LLM** (Stage 5) —
   which contradicts the "cut the LLM from cognition" direction. Replace the trigger:
   mine the **Phase-3 episode log** for recurring action subsequences that *reliably reduce
   prediction error or achieve a subgoal*, and chunk those into a skill. Discovery driven
   by demonstrated predictive/control value, not by gap-language and not by the LLM.
3. **5C — Hierarchy + skill-composed planning (the genuinely new part).** Grep confirms
   **no** option/macro/sub-skill composition exists today — skills are flat. Add: a skill
   may call primitives *or other skills*; it predicts its *aggregate* outcome; error at the
   skill level updates the skill, error at the primitive level updates primitives (two
   levels, one spine). Point `planning/goal_planning.py` (already does subgoals) at the
   grounded skill library so plans compose skills that have earned their predictions.
4. **5D — Upgrade atrophy from usage-based to value-based.** `maybe_integrate_or_atrophy`
   already prunes ("used in ~100 cycles? else removed"). Replace the *criterion*: prune on
   **predictive/control value**, not bare usage — the same law that prunes ungrounded
   concepts (#1's regulation, applied to the library so it doesn't bloat with dead
   options).

**The curriculum this creates:** ground primitives (Phase 3) → discover skills (5B) →
compose meta-skills (skills that call skills). This bottom-up curriculum *is* the
data-efficiency mechanism — each grounded layer makes the next cheaper to learn.

**Connection to LM-as-mouth (Phase 2):** skill names become grounded concept handles the
thought object (2A) can reference, so he can authentically say "I tried X and it failed"
where X traces to a real grounded skill, not a borrowed word.

**Success metric (skill-level transfer):** a learned skill transfers when it achieves its
subgoal in a **novel context it was not discovered in** — measured like Phase 3B but at
the option level. That transfer is the evidence the hierarchy is real, not memorized.

**Exit:** a growing library of grounded, reliability-weighted skills the planner composes
into plans; demonstrated skill-level transfer; pruning keeps the library honest.

**Effort:** substantial, research-adjacent (weeks→months). **Risk:** high — inherits
Phase 3's research uncertainty and adds the option-discovery problem (chunking is itself
unsolved in general). **World/Phase dependency:** gated on Phase 3 transfer + 4A. The
*schema* (5A) can be drafted earlier, but discovery (5B) needs grounded primitives to
chunk.

## Dependency graph & sequencing

- **Parallelizable now:** Phase 1A, 1B, and Phase 2 (2A→2B→2C) are independent and can
  run concurrently. None depend on the world.
- **Phase 3 depends on:** nothing in 1/2 strictly, but should land *after* 1A (so the
  world loop's new content is sealed by the membrane law from birth) and *after* 1B (so
  the surprise/curiosity signals the loop drives are homeostatically regulated).
- **Phase 4 depends entirely on Phase 3's verdict.** Do not pre-build it.
- **Phase 5 depends on Phase 3 transferring (the 4A branch).** Its *schema* (5A) can be
  drafted in parallel, but skill *discovery* (5B) has nothing to chunk until grounded
  primitives exist. Phase 5 is where 4A's "expand grounding" actually gets its structure —
  and where the low-data/priors risk gets its real mitigation (learned skills = priors).

**Single highest-leverage move (direction doc):** Phase 3A. Everything ambitious rides on
it; everything in Phase 1/2 is worth doing regardless.

## The honest metrics

- **Stability metric (Phase 1):** hours-before-a-human-must-fix-something → target
  indefinite. Measures that he isn't drifting.
- **Grounding metric (Phase 3):** transfer-above-chance on unseen commands. Measures that
  he is *thinking*, not just running. These are different metrics; do not let a green
  stability number be mistaken for grounding.
- **Competence metric (Phase 5):** skill-level transfer — a discovered skill achieving its
  subgoal in a context it wasn't discovered in, and the library compounding (later skills
  needing fewer episodes to ground). Measures that experience is *accumulating*, not just
  grounding one action at a time.

## What NOT to do (guardrails carried from the direction doc)

1. Don't build the idleness cost (#3) before the world loop — it will thrash.
2. Don't feed internet text to the native LM or use it as a grounding source.
3. Don't retire authored symbols before a grown concept out-predicts them.
4. Don't force every signal to baseline — regulate around shifting setpoints (allostasis).
5. Don't treat Phase 3 as a feature to ship — it's an experiment that may return "no," and
   that answer is the point.
6. Don't let the external LLM back into the cognition path — it stays tool-only; the goal
   is to cut it from cognition entirely.
7. Don't hand-author skills into the library (Phase 5) — they must be *discovered* from
   grounded experience, or they're ungrounded, same problem as authored symbols.
8. Don't let the skill library bloat — it needs its own pruning (5D); an unbounded option
   set is a new runaway, just at the skill layer.
