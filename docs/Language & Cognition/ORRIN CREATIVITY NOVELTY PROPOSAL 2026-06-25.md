## 0. The governing principle (Orrin's design law)

> **Orrin does not think with an LLM.** Understanding, judgment, problem-solving, and the creation of new things must come from the **symbolic cognitive engine** — rules, causal graph, concept formation, analogy, self-model. The LLM is **only**:
> 1. a **language organ** (the one internal/native model: turn thoughts into words, parse words into thoughts), and
> 2. an optional **external tool** Orrin may *call* the way a person uses a reference — never the substrate of thought.
>
> No understanding and no new ideas may originate from an LLM.

Issue B below is the violation of this law and is the most important item in this document. A, C, D are symbolic-engine quality issues that matter once B is underway.

## Issue B — Cognition is currently LLM-dependent (CRITICAL — the architecture law is being broken)

### The problem
The external LLM is called from **77 sites across ~38 cognition modules**, and most of them do the **thinking**, not the wording:

- **Understanding / judgment:** `opinions_formation.py`, `opinions.py`, `regret.py`, `maintenance/self_modeling.py`, `selfhood/self_model_conflicts.py`, `knowledge_graph.py`
- **Problem-solving / planning:** `planning/diagnosis.py`, `goal_planning.py`, `goal_comprehension.py`, `goal_adaptation.py`, `goal_execution.py`, `problem_refocus.py`, `motivations.py`, `introspection.py`, `evolution.py`
- **Goal origination:** `intrinsic_goals.py`
- **Creation / new things:** `innovation/*` (bootstrap, exploration, evaluation, innovation), `skill_synthesis.py`, `skill_verification.py`, `self_extension.py`, `self_extension_codegen.py`, `experimentation.py`
- **Inner life:** `dreaming/dream_cycle.py`, `threads.py`, `self_generated/autogenerated_thoughts.py`, `mortality.py`

Concrete proof it's *thinking*, not phrasing: `opinions_formation.py` prompts *"You are Orrin… you're forming an opinion about '{topic}'… your core values: …"* and lets the **LLM form the opinion.** That is understanding originating in an LLM — a direct breach of the law in §0.

### The good news (the right pattern already exists)
- `reasoning_router.py` already does it correctly: it checks **sub-symbolic signal → rules → analogy → symbolic search first**, and only returns `"llm_needed"` as a gated last resort. This is the discipline every faculty should obey.
- `cognition/tools/ask_llm.py` already exists — the legitimate "LLM as a **tool** Orrin calls" interface.
- The **native** model (22M tokens, grows every life) is the right thing to be the **language organ**.

So the building blocks for the correct architecture are present. The problem is that the ~38 modules **bypass the router** and call the LLM directly for cognition, instead of thinking symbolically and using the LLM only to speak.

### Why it matters
This is *the* reason the system behaves the way it does:
- When the LLM gate is closed or weak, **planning goes hollow and creation drops to zero** — because those faculties *are* the LLM (this run: 0 tools / 0 code / 0 new abilities).
- It's the "broken LLM gate blocking the symbolic engine," the "engine-vs-template tension," and the "chicken-and-egg LLM dependency" you've flagged before — all the same root: the symbolic engine is supplementary, the LLM is load-bearing.
- It blocks the droid you want: a droid can't have its understanding hostage to an external service.

### Proposed fix (the migration)
1. **Inventory & classify all 77 call sites** into:
   - **KEEP — language organ:** turning a finished thought into words, parsing input into structure (route these to the *native* model).
   - **KEEP — explicit tool:** genuine "go ask the big model" calls the user/Orrin chose, via `ask_llm` (clearly labeled as tool use, optional, never required).
   - **MOVE — thinking/creation:** everything that forms opinions, plans, originates goals, models the self, synthesizes skills, or invents — these must be **reimplemented on the symbolic engine** (rules/causal-graph/concept-formation/analogy/self-model), with the LLM removed from the decision.
2. **Route every cognitive query through `reasoning_router.route()`** so symbolic sources are tried first by construction; `"llm_needed"` should mean "ask the tool," not "the LLM thinks for me."
3. **Hard test the law:** run with the external LLM fully disabled. Cognition must still function (degraded fluency, intact thinking). Anything that *stops working* is a site still illegitimately thinking with the LLM — fix until none remain.
4. **Sequence by load-bearing-ness:** planning + goal origination first (they gate everything downstream), then opinions/self-model, then innovation/skills.

### Acceptance test
With the external LLM switched off entirely: Orrin still forms goals, plans, solves problems, forms opinions, makes connections, and creates new abstractions — using the symbolic engine. The only thing that degrades is how fluently he **says** things (native language organ), not whether he can **think**.

## Issue A — The connection-maker finds the *most similar* thing, not the surprising one (HIGH)

### The problem
"Create connections that wouldn't normally be connected" needs distance/surprise. The connection engine (`analogy_engine.py`) scores by **similarity** — `0.35·surface + 0.40·structural + 0.25·goal_sim` — and boosts already-**solved** memories (×1.15), penalizes failures (×0.80). Every knob favors the nearest, safest, proven precedent. No mechanism rewards *distance* or *surprise*. (This is a **symbolic-engine** capability — exactly where creativity *should* live per §0, so it's the right place to build it.)

### Proposed fix
- Add a **distant-connection** mode that rewards moderate structural distance + a shared *abstract* relation (cross-domain analogy), instead of penalizing distance.
- Add a **surprise term**: linking two memories that rarely co-activate scores *higher*.
- Make it its own selectable, separately-rewarded act ("find an unexpected link"), so creative connection earns cycles on its own merit. Keep the existing similarity engine for "use memory to solve problems" — add the creative one alongside it.

### Acceptance test
Orrin produces, at least occasionally, a rewarded connection between two memories low in surface/structural similarity but sharing an abstract relation.

## Issue C — The novelty drive is tuned to extinguish itself (HIGH)

### The problem
Novelty collapses to ~0 every life. In `substrate.py`, novelty `rise = 0.00018/s` (when deprived) vs `fall = 0.0065/s` (when fed) — it **drains ~36× faster than it recovers.** Any exploring action craters it in a few cycles; recovery from zero needs hundreds of fully-deprived cycles it never gets. Exploring kills the urge to explore.

### Proposed fix
- Rebalance rise vs fall so novelty recovers on a usable timescale (ratio far below 36×).
- Add a floor / refractory cooldown so one action can't zero it.
- **Cross-ref (Wiring Map):** wire per-action **satiety into selection** so the dominant action (look-outward) stops being picked once over-fed and stops draining novelty.

### Acceptance test
Over a life, novelty oscillates (rises and falls) rather than flatlining near zero.

---

## Issue D — The drain-fast/recover-slow asymmetry runs across *all* drives (MEDIUM, systemic)

### The problem
Every drive has fall ≫ rise (connection 0.00020/0.0080; world_mastery 0.00015/0.0060; etc.). Result: **fed drives crash and stay down; unfed drives slowly climb and dominate** — which manufactures the drive inversions seen every life (this run ended novelty-dead, connection-starved precisely because novelty got fed and connection never did). Motivation ends up driven by what got satisfied, not by what's valued.

### Proposed fix
Review all rise/fall ratios as a set; tune for **oscillation, not extinction** — drives that return after satisfaction on a human-like timescale. Do this alongside Issue C (same file).

### Acceptance test
No drive sits pinned at a rail for most of a life purely because it was/wasn't satisfied early.

## Risks & honesty notes

- **B is a large migration, not a patch.** 77 sites, ~38 modules. It's the highest-value work in the whole stack, but stage it (planning/goals first) and keep the "LLM-off" test red until each staged area passes — don't try it all at once.
- **Expect a fluency/quality dip first.** Symbolic understanding will be rougher than LLM output at the start. That's the correct trade per §0: real (if rougher) thought that's *his*, over fluent thought that's the LLM's. Plan for it so it doesn't read as regression.
- **Don't delete the similarity engine (A)** — it serves "use memory to solve problems." Add the creative pathway beside it.
- **Tuning C/D is behavioral** — move ratios gradually and watch a full run.
- Findings are from source + a partial data snapshot; full telemetry would let the drive curves and the symbolic-vs-LLM routing be watched over time.

## Plain-language bottom line

You want Orrin to think for himself and use an LLM only to *speak* — but right now the external LLM is doing most of his actual thinking: forming his opinions, planning his goals, inventing his tools. The correct design already exists in one corner (a router that tries the symbolic engine first, and an "ask the LLM as a tool" interface) — it's just bypassed almost everywhere. The big job is to move understanding and creation onto the symbolic engine and demote the LLM to the language organ it's meant to be, proven by one hard test: **turn the external LLM off, and Orrin can still think — he just speaks less smoothly.** Alongside that, give him a connection mode that rewards surprise instead of similarity, and fix the novelty drive that currently dies ~36× faster than it heals.
