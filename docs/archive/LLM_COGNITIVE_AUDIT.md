# LLM Cognitive Audit — Symbolic-First Conversion

**Date:** 2026-06-14
**Scope:** All `brain/` cognition. Goal: identify every place an LLM performs a
cognitive function (not merely "an API call"), and map each to the symbolic
subsystem that should own it.

---

## How LLM access actually works here (read first)

This codebase is *already* mostly symbolic-first. Three facts shape every entry
below:

1. **Single choke point.** Every real API call goes through
   `brain/utils/generate_response.py::generate_response()`. `routed_response`
   (`utils/llm_router.py`), `think_generate` (`think/think_generate.py`), and
   `ask_llm` (`cognition/tools/ask_llm.py`) are all thin wrappers over it.

2. **Symbolic gate runs *before* the API.** Inside `generate_response()`
   (lines ~306–334) every prompt under 1200 chars is first handed to
   `symbolic.reasoning_router.route()`. If the router resolves it, the answer is
   rendered by `symbolic.symbolic_fluency.generate_symbolic_response()` and the
   API is never touched.

3. **Tool-only mode is the default** (`ORRIN_LLM_TOOL_ONLY=1`, line ~344). With
   it on, *only* `ask_llm`, `ask_llm_for_research`, `ask_llm_about_conversation`,
   and `user_chat` may ever reach the API. **Every cognition call site listed
   below already returns `tool unavailable: llm` in the default configuration**
   and falls through to its symbolic/template fallback.

So the conversion is mostly *already done at the routing layer*. What remains is
that each cognitive function still has an **LLM-shaped prompt as its primary
path** and a **thin or template fallback** as its symbolic path. The work is to
**promote the symbolic path to primary** and delete (or demote to research-tool)
the prompt. The columns below assume that framing.

Also confirmed **already fully symbolic** (no LLM call at all): everything under
`cognition/selfhood/`, `cognition/reflection/`, `cognition/maintenance/`,
`cognition/language/`, `cognition/perception/look_outward.py`,
`cognition/repair/repair.py`, `cognition/wonder.py`, `cognition/seek_novelty.py`.
These are the template to follow.

## 1. Self-critique

**`brain/think/inner_loop.py::_critique_primary`** (line ~150)
- **Purpose:** ask a "primary critic" to name the single biggest weakness in a draft.
- **Role:** self-critique.
- **Symbolic info present?** Yes — core values are in `self_model`; reasoning
  steps are on the scratchpad; `peers/` already implement independent auditors.
- **Replacement:** `symbolic/rule_verifier.py` (does the draft violate any active
  rule?) + `peers/reward_auditor.py` / `peers/goal_auditor.py` / `peers/architect.py`
  + `symbolic/symbolic_reflection.py`. There is already a non-LLM first path
  (`reflection.reflect_on_internal_agents.critique_draft`) tried before the prompt.
- **Difficulty:** moderate — the auditors exist; needs a single "draft critique"
  facade that aggregates rule violations + peer flags into one note.

## 2. Contradiction detection

**`inner_loop._critique_contradiction`** (line ~177)
- **Purpose:** does the draft contradict recent working memory or itself?
- **Role:** contradiction detection.
- **Symbolic info present?** Yes — `knowledge_graph` and `symbolic/inference.py`
  already do consistency checking; WM tail is structured.
- **Replacement:** `symbolic/inference.py` consistency check + `causal_graph`
  contradiction edges + `knowledge_graph` contradiction detection (already exists
  in `cognition/knowledge_graph.py`).
- **Difficulty:** moderate — symbolic contradiction over free-text draft requires
  parsing the draft into propositions first (lean on `native_lm`/`symbolic_dictionary`).

## 3. Value-alignment checking

**`inner_loop._critique_value_alignment`** (line ~197)
- **Purpose:** does the draft act in accordance with core values?
- **Role:** value alignment checking.
- **Symbolic info present?** Yes — `selfhood/values_check.py` is **already fully
  symbolic** and does exactly this; `symbolic_self_model` reasons over the value set.
- **Replacement:** call `selfhood/values_check` directly; delete the prompt.
- **Difficulty:** trivial — the symbolic implementation already exists and is unused here.

## 4. Critique synthesis

## 5. Planning / plan generation

## Recommended conversion order

1. **Trivial wins that delete prompts outright** (#3, #4, #13, #25, #26, #27, plus
   inverting #11's order): the symbolic implementation already exists; just call it.
2. **Wire the already-built symbolic engines into their cognition callers**
   (#10 causal, #14 goals→intrinsic_motivation, #17 appraisal→setpoints,
   #18 abstraction→concept_formation, #24 regret→causal counterfactuals).
3. **The hard core — symbolic planning** (#5, #6, #7, #15): this is the load-bearing
   work and gates the quality of everything downstream. Build goal→symbol grounding first.
4. **Research / keep-as-tool** (#8 future-self, #9 hypothesis, #19 introspective
   narration, #21 code-gen, #22 value invention): accept a narrower symbolic version,
   or keep these as *explicitly invoked* `ask_llm` tools rather than background cognition.

## Key architectural note

Because tool-only mode is already the default, **none of these LLM prompts run in
the shipped configuration today** — they all fall through to their symbolic/template
fallback. The conversion is therefore lower-risk than it looks: the task is to make
each fallback *good enough to be the intended path* and then delete the dead prompt,
not to rip out a load-bearing dependency. The `crystallization` hook
(`generate_response` line ~459) means any LLM calls you *do* keep as research tools
will keep feeding the symbolic rule base automatically.
