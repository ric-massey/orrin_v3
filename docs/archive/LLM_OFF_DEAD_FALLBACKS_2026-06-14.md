# LLM-Off Audit: Dead Symbolic Fallbacks in the Default Tool-Only Deployment

**Date:** 2026-06-14
**Scope:** Find any code that fails to "turn on" (degrade to symbolic) when the LLM is off for cognition.
**Status:** Findings only — no code changed yet.

---

## TL;DR

Orrin runs **tool-only by default** (`ORRIN_LLM_TOOL_ONLY=1`): the LLM is a registry tool
that only allowlisted callers (`ask_llm`, `ask_llm_for_research`,
`ask_llm_about_conversation`, `user_chat`) may reach; all cognition is meant to stay
symbolic. The project already built the correct predicate for this —
`llm_callable_by(caller)` in `brain/utils/llm_gate.py:81` — precisely because
`llm_available()` *"ignores tool-only mode and over-reports."*

A large set of cognition modules still gate their rule-based / symbolic fallback on
**`llm_available()`** instead of **`llm_callable_by()`**. In the intended default
deployment (API key present so the LLM *tool* works for chat, while cognition is
tool-only) `llm_available()` returns **`True`**. The result:

```python
if not llm_available():        # False in default deployment → branch SKIPPED
    <rich symbolic fallback>   # DEAD CODE in normal operation
else:
    raw = llm_ok(generate_response(..., caller="<not allowlisted>"))  # → "tool unavailable"
    # raw is None/"" → degenerate output; the good fallback right above never runs
```

These modules don't degrade the way they were written to. They call the LLM, get
*tool unavailable*, and emit degenerate output instead of the symbolic fallback
sitting directly above them.

> **Precondition:** This bites whenever `OPENAI_API_KEY` is present **and**
> `ORRIN_LLM_TOOL_ONLY` is on — i.e. the *intended* default deployment, where the
> LLM tool serves `user_chat`/`ask_llm` but cognition stays symbolic. With no key at
> all, `llm_available()` is `False`, the fallbacks fire correctly — but then the LLM
> tool is dead too.

---

## Root cause

`llm_available()` (`brain/utils/llm_gate.py:42`) answers *"is the LLM tool reachable
at all?"* — config enabled + key present + breaker closed + router importable. It does
**not** account for tool-only mode.

`llm_callable_by(caller)` (`brain/utils/llm_gate.py:81`) answers the question cognition
actually needs: *"could THIS caller reach the API right now?"* — i.e.
`llm_available()` **and** (tool-only off **or** caller is allowlisted).

Gating a symbolic fallback on `llm_available()` is therefore wrong in the default
deployment: the fallback is keyed to a condition that is effectively never true.

| Location | Behavior with cognition-LLM off (default deployment) |
|---|---|
| `brain/cognition/regret.py:173` | Skips the rule-based *"I notice I X. I'm not sure that was right."* reflection; returns the string `"no regret reflection generated"` and **stores nothing** to working/long memory. |

## Separate issue — forward simulation: not gated at all, fails *open*

`brain/think/simulate.py` (`simulate_lookahead`, `run_debate`) has **no gate and no
symbolic path**. With cognition-LLM off it returns:

```python
{"steps": [], "projected_state": "", "positive": True, "confidence": 0.5, "branches_explored": 0}
```

`brain/think/meta_controller.py:229` `simulate_outcome` reads that and returns
`positive or conf < 0.4` → **always `True`**. So the "simulate before you act" check
silently passes **every** action in the default deployment, while still burning ≥1
blocked LLM round-trip per cycle.

Note: `inner_loop`'s own `run_debate` calls are fine — they sit behind a
`llm_callable_by("inner_loop")` gate that routes to the symbolic inner loop or an
honest defer (`brain/think/inner_loop.py:419`).

## Also affected (lower priority)

- `brain/symbolic/llm_gate.py:106` `gated_generate` calls `generate_response(prompt,
  caller=caller)` with the passed-in (non-allowlisted) caller. It gates on
  emotion/conflict/concurrency but **not** on `llm_callable_by`, so for cognition
  callers it returns `""` in tool-only mode. Consumers (`self_reflection`,
  `self_modeling`) mostly have a prior symbolic attempt, so impact is limited — but
  the call is a wasted, always-blocked round-trip.
- `brain/cognition/skill_synthesis.py:458,626` genuinely require the LLM (code
  generation). They gate on `llm_available()` so in tool-only mode they do a wasted
  call and return a less-clean error rather than skipping cleanly. Cosmetic.

## Verified-correct for contrast (these degrade properly)

- `brain/think/inner_loop.py` — `llm_callable_by` → symbolic inner loop or honest typed defer.
- `brain/behavior/speech_pipeline.py` `build_response` — symbolic compose → LLM → rule fallback; always returns non-empty. User-facing speech is safe.
- `brain/cognition/theory_of_mind.py` — pure rule-based, no LLM dependency.
- `brain/utils/embedder.py` `get_embedding` — local sentence-transformers; memory store/retrieve works fully offline.
- `brain/cognition/planning/pursue_goal.py` `_generate_plan` — falls back to `_symbolic_plan` (and honest `needs_capability` blockage).
- `brain/cognition/planning/goals.py:504` `decompose` and `brain/cognition/comprehension.py:55` `_parse` — land on their fallback either way; only cost is a wasted round-trip.

---

## Recommended fix

1. **Mechanical, low-risk:** at each dead-fallback gate site, replace
   `if not llm_available():` with `if not llm_callable_by("<that caller>"):` so the
   symbolic branch fires whenever cognition can't actually reach the API.
   - `regret.py:173`, `opinions.py:717` & `:961`, `goals.py:532`,
     `autogenerated_thoughts.py:118`, `introspection.py:109`, `self_reflection.py:224`
     (and optionally `:84`).
2. **Needs a design decision:** `simulate.py` / `meta_controller.simulate_outcome` —
   either add a `llm_callable_by` gate that returns an honest "skip simulation"
   (don't fail-open to `positive=True`), or give it a symbolic lookahead path.
3. **Optional cleanup:** swap the wasted-round-trip sites
   (`goals.py:504 decompose`, `comprehension.py:55`, `skill_synthesis.py`,
   `gated_generate`) to `llm_callable_by` to avoid blocked calls each cycle.
