## 1. Phase 1 — Clean the candidate pool  *(low risk, reversible)*

### What changed

**`brain/think/think_utils/select_function.py`**
- New shape-based selectability filter after `_ALWAYS_EXCLUDE`:
  - `_NON_SELECTABLE_PREFIXES` — `explore_` + upkeep/accessor prefixes
    (`apply_/update_/compute_/…/get_/set_/has_/should_`).
  - `_SELECTABLE_PREFIX_EXCEPTIONS = {"update_world_model"}` — the one real behavior
    that starts with a denied prefix.
  - `_NON_SELECTABLE_EXACT` — trivial-name leaks (`available`, `get`, `commit`, …) and
    internal reward/calibration calc (`calibrated_reward`, `check_and_reward*`, …).
  - `_is_selectable_name(name)` — exact denials and the exception are checked **before**
    the prefix sweep, so `update_world_model` survives while the plumbing it resembles is
    dropped.
  - Per the plan's caution, `is_*` / `maybe_*` are **not** denied as a class (would have
    killed `maybe_form_opinion` etc.); confirmed plumbing goes in `_NON_SELECTABLE_EXACT`.
- Wired `_is_selectable_name` into **both** pool builders: `_load_action_defs` (dict and
  str branches) and `_load_actions`.

**`brain/registry/cognition_registry.py`**
- `persist_names` now filters `explore_*`, so corrupted stubs can never be written into
  `cognitive_functions.json` even if they leak into discovery.

**`brain/behavior/behavior_generation.py`** — §2.4 root cause
- *(A token-dedupe + length-cap of the `explore_<topic>` name was first added here as the
  §2.4(a) containment. Phase 5 / Option A then removed the stub-generation entirely — no
  `.py` file is written at all — which is the **complete** §2.4 cure and makes the
  name-construction code unnecessary; it was removed. See §4.)*

**`brain/cognition/self_generated/autogenerated_thoughts.py`** — one-time cleanup
- Truncated from 1547 → 179 lines: the 76 generated `explore_*` stubs were removed, the
  hand-written `maybe_generate_thought` / `_generate` (called directly from
  `finalize.py:439`, **not** via the selector) preserved intact.
- `cognitive_functions.json` regenerated by re-importing the registry.

### Acceptance — met

```
pool = 147   (plan estimate ~120–140; 147 real behaviors survive)
explore_* in pool = 0
maybe_generate_thought still importable = OK
```

**"No behavior lost" — proven, not asserted.** Cross-referencing the 173 dropped names
against `decision_stats.json`, only **3** were *ever* chosen historically:
`explore_research_a_real_topic_and_write_what_i_find` (×4) and
`explore_write_a_cognitive_function_or_tool_that_improves_something` (×4) — both the junk
being removed — and `check_and_reward_goal_closure` (×1, an internal reward calc). No real
behavior was ever productively selected and then dropped.

### Note on §2.4(b)

The plan's §2.4(b) concern (generated stubs being auto-discovered into the cognition pool)
is now **fully resolved by Phase 5 / Option A**: no stub file is written at all, so there is
nothing to relocate or discover. The `persist_names` `explore_*` filter, the in-selector
shape filter, and the one-time truncation remain as defense-in-depth.

### Rollback
Pure subtraction from the pool. Revert the `_NON_SELECTABLE_*` block + the three
`_is_selectable_name` call sites and the `persist_names`/`behavior_generation` edits. No
state migration.

## 2. Phase 2 — Real exploration with safety  *(medium risk, revertible via flag)*

### What changed (`select_function.py`)

1. **`_SAFE_TO_EXPLORE`** — a frozenset of ~55 reversible, internally-scoped functions
   (procedural reads/observations + reflect/recall/detect/consolidate/dream), **curated
   against the live 147 pool** and verified `⊆ pool`. The irreversible "situational, leave
   rare" tail (`abandon_goal`, `mutate_directive`, `evolve_core_value`,
   `emergency_self_modification`, `submit_finetune_job`, `run_*experiment*`,
   `propose_value_revision`, goal create/abandon, etc.) is explicitly excluded and
   documented inline — those stay selectable **on merit** but are never **force-sampled**
   (E4: rarely-used ≠ safe-to-try).

2. **ε-greedy gated sampling** inserted right after the argmax pick and **before** the
   threat-arbiter block. With probability ε it replaces the argmax with a **softmax draw
   over the safe, rarely-used tail** (count < 8, not in `recent`). ε defaults to 0.10, can
   be raised by `exploration_drive`, and is capped at 0.30. Setting
   `context["_exploration_epsilon"] = 0.0` disables the branch entirely (documented
   rollback) regardless of drive. Because it sits before the arbiter, a real reflex spike
   still overrides whatever exploration picked.

3. **Bandit-hint normalization fixed** (`_bandit_hint_scores`): min-max replaced with a
   **fixed-scale clamp** `max(0, min(1, v))`. Min-max destroyed the bandit's cold-arm
   optimism — when every candidate is cold they all score `1.0`, the span collapses to 0,
   and they all normalized to `0.0`. The clamp preserves the optimistic `1.0` as a real
   positive hint. Confirmed: cold arms now read `1.0` instead of `0.0`.

4. **`w_band` 0.15 → 0.25** so the (now meaningful) bandit hint can compete with the
   additive boosts. Still a hint, not the decider — the ε-branch is what guarantees trials.

### Acceptance — safety verified

- **Force-sample safety (acceptance #3):** across **2,400** forced-exploration calls
  (ε=1.0, high drive, varied attention modes), **every** ε-sampled pick was in
  `_SAFE_TO_EXPLORE`; no exception; no out-of-allowlist target.
- **No double-execution (acceptance #4):** `_ALWAYS_EXCLUDE` names (incl.
  `pursue_committed_goal`) are not in the pool, so they cannot be ε-sampled — confirmed.
- **Breadth / reward-health (acceptance #1, #2):** these require a multi-hundred-cycle
  staging run with a live reward signal and a pre/post `decision_stats.json` +
  `reward_trace.json` diff. The mechanism is in place; **run the staging diff before
  merge** to confirm `avg_reward` does not drop and distinct-fn count rises.

### Rollback
Set `_exploration_epsilon = 0.0` (kills the ε-branch, reverts to argmax) without removing
code. The `w_band` and normalization changes revert independently.

## 3. Phase 3 — Goal → capability recruitment  *(medium/high risk)*

### 3a — Executive lane: semantic `recognise_step_action` (`step_execution.py`)

Rewritten to three tiers: (1) literal tool name, (2) habitual keyword rules, (3) **semantic
fallback over `_PROCEDURAL_FNS` only**, with a confidence floor below which it still returns
`None` (genuinely internal steps stay deliberative — System-1/System-2 split preserved).

- `_semantic_step_match` matches the step against the **curated capability descriptions**
  (E5 fix — not raw docstrings), via stopword-filtered keyword-overlap so it ships with **no
  model dependency**; an embedding backend can replace the similarity call later without
  changing the interface.
- `_SEMANTIC_FLOOR = 0.22` (set to `2.0` to disable → reverts to the 8-rule table).

**Safety (Dig#7 / §4.5):** the match space is bounded to `_PROCEDURAL_FNS` (reversible
reads/observations) and `execute_step_action` already refuses non-procedural fns under
`_procedural_only`, so broadening 3a **cannot** recruit an irreversible act or double-execute
the deliberate lane. Verified: across a battery of step texts, every return is
**procedural-or-`None`** — research/observe/encyclopedia steps map to
`research_topic`/`look_outward`/`wikipedia_search`; reflect/consolidate/decide steps return
`None`.

### 3b — Deliberate lane: goal-derived recruitment (`select_function.py`)

A new `_goal_recruit` boost is computed from the **active goal's own title/description/tags**
against the curated capability descriptions (capped at +0.40, comparable to `s_attn`), added
to the score as `s_goal_recruit`. This replaces reliance on the static goal-pursuit
name-lists for goal-specific routing.

**Acceptance — verified:** distinct goal *types* recruit visibly distinct function sets:

| Goal text | Top recruited |
|---|---|
| "research the history of ancient rome and read sources" | `research_topic`, `fetch_wikipedia`, `read_a_book` |
| "reflect on my own beliefs values and who i am becoming" | `reflect_on_self_beliefs`, `reflect_on_think`, `meta_reflect`, `detect_contradictions` |
| "seek out something new and unexplored to discover" | `seek_novelty`, `generate_concepts_from_memories`, `generate_intrinsic_goals` |

(An initial pass mis-ranked the research goal because shared stopwords like "the"/"and"
inflated overlap; fixed by adding the stopword-filtered `_capability_overlap`, used by both
3a and 3b, rather than touching the shared `_kw_overlap_score`.)

### 3c — Curated capability text (new file)

`brain/data/capability_descriptions.json` — `{fn_name: "goal-prose description"}` for the
16 `_PROCEDURAL_FNS` (Phase 3a) plus a broad goal-recruit subset (reflection / self-model /
goal / creative / reading), loaded by the cached `_capability_descriptions()` helper. This
is the E5 fix and the seed of Phase 4. Additive data — safe to leave on rollback.

### 3d — E6 dead-`pursue_committed_goal` cleanup

`pursue_committed_goal` is in `_ALWAYS_EXCLUDE` (it runs in the Executive lane), so every
scoring branch keyed on it was dead. All **live** branches were fixed:

- alert / engaged attention-mode literal lists — dead name removed.
- deadline `_tension_boost` — dead `pursue_committed_goal` lines removed (urgency now lands
  on the real `assess_goal_progress` / `plan_next_step`).
- `_GOAL_PURSUIT_FNS` no-goal suppression set — dead name removed.
- `_ADAPT_ACTION_FNS` — dead name removed.
- **Goal-shielding bug fixed:** `if _has_committed_goal and "pursue_committed_goal" in
  actions:` was always `False`, so the Shah/Friedman/Kruglanski (2002) shielding **never
  fired**. Now gated on `_has_committed_goal` alone — exploration reads are damped during
  goal pursuit as originally intended. *(Behavioral change — see §5.)*
- **Commitment bias relocated:** `if name == "pursue_committed_goal": total +=
  _commitment_bias` was unreachable; moved to **`attend_goal`** (the thin selectable
  "consciously decide to focus" proxy, which IS in the pool), so the will/commitment
  follow-through bias now actually applies.

No `name == "pursue_committed_goal"` scoring branch remains. Residual references are the
authoritative `_ALWAYS_EXCLUDE` entry, explanatory comments, and harmless data-dict entries
in `_SEMANTIC_PRIORS` / `_neuro_boost` / `_emo_mode_map` / `_USER_HELPFUL_FUNCTIONS` /
`_EXECUTION_FNS` — never scored (the name isn't in the pool), and explicitly Phase 4's
tag-ification scope. Left in place to keep the diff bounded to E6's enumerated lines.

### Rollback
`_SEMANTIC_FLOOR = 2.0` disables the semantic fallback (reverts to the 8-rule table); revert
the `_goal_recruit` block + `s_goal_recruit` term to undo 3b. `capability_descriptions.json`
is additive data, safe to keep.

## 4. Phase 5 — Sandbox the self-written-code path  *(security; Option A)*

### Finding (confirmed)

The action-gate `execute_python_code` handler (`action_gate.py:1013-1028`) ran model-/auto-
generated code via a **bare in-process interpreter call with full builtins** — no AST check,
no subprocess, no timeout, no resource limit. Tracing its only producer showed it was fed
**solely** by `behavior_generation.py`'s auto-generated-stub pipeline, and those stubs only
ever **appended one thought to working memory**. (The separate `toolkit.execute_python_code`
registration already runs in an isolated subprocess via `think/sandbox_runner.py`; the
`exec` calls in `cognition/skill_synthesis.py:189,196` live inside a harness string executed
by that subprocess runner — both are off the dangerous path. Verified.)

### Decision: Option A (the plan's recommendation)

Because the only feeder just appended a thought, **no code-execution capability was ever
needed** for it. Implemented Option A and hardened the handler so a bare in-process exec can
never run again:

- **`behavior_generation.py`** — removed the synthesized `.py` stub, the `write_file`
  proposal that wrote it, and the `execute_python_code` proposal that ran it. Replaced with a
  single structured **`append_thought`** proposal carrying the same content. This also
  **fully moots §2.4** (no stub file is written, so no corrupted `explore_*` defs can be
  generated or auto-discovered). Removed the now-unused `import json` / `AUTOGENERATED_THOUGHTS`
  / `ROOT_DIR`.
- **`action_gate.py`** — new `append_thought` handler (structured working-memory write). The
  `execute_python_code` handler no longer execs in-process: it is **disabled by default**
  (`ALLOW_CODE_ACTIONS` unset → rejected) and, only when explicitly enabled, routes through
  the hardened subprocess sandbox **`run_python_sandboxed`** (AST allowlist + POSIX rlimits +
  `-I` subprocess + wall-clock timeout). This also wires up the previously-dead sandbox
  (plan step 6 — "dead safety code is a trap").
- **`behavior/tools/sandbox.py`** — fixed a latent cross-platform bug surfaced by wiring it
  up: `preexec_fn` set `RLIMIT_AS`, which **macOS rejects**, aborting the subprocess launch.
  Each rlimit is now applied best-effort/independently, so an unsupported limit is skipped
  while CPU/FSIZE (where supported) + `-I` isolation + the wall-clock timeout still bound the
  child. *(On macOS the 256 MB address-space cap is not enforced; the AST allowlist + timeout
  remain the primary guards. On Linux all three rlimits apply.)*

### Acceptance — met (`tests/brain/test_code_action_sandbox.py`, 11 tests)

- **No in-process exec remains:** `grep -nE "\bexec\s*\(" action_gate.py` → none; a test also
  asserts no `exec(`/`eval(`/`compile(` call in the module.
- **Hostile code blocked:** `import os`, `from os import path`, `open(...)`, `__import__(...)`,
  `eval(...)`, `exec(...)` are all rejected by the AST allowlist **before** any execution.
- **Infinite loop terminates:** `while True: pass` returns `status="error"` well under the
  in-process hang failure mode (CPU rlimit / wall-clock timeout), not a hang.
- **Gated + sandboxed:** handler checks `ALLOW_CODE_ACTIONS` and delegates to
  `run_python_sandboxed`; behavior_generation emits `append_thought`, never
  `execute_python_code`/`write_file`.

### Rollback

Option A is data-only and reverts by restoring the prior proposals. The handler's
default-off `ALLOW_CODE_ACTIONS` gate means code actions stay disabled unless explicitly
opted in.

## 6. Behavioral changes a reviewer should know

1. **Goal-shielding now actually fires** (E6 fix). Previously a no-op; now exploration reads
   (`look_outward`, `seek_novelty`, `search_own_files`, …) are scaled ×0.4 while a goal is
   committed. This is the intended design but is a *live* behavior change — watch that it
   doesn't over-suppress exploration in combination with Phase 2 (Phase 2's ε-branch is not
   affected by the shield; only the standing outward boost is).
2. **Commitment bias now applies to `attend_goal`** instead of an unreachable name.
3. **`w_band` 0.25** raises the bandit's influence on the deliberate pick.
4. The pool shrank 287 → 147; any external dashboard counting "available functions" will
   reflect this.
5. **Code actions are off by default** (`ALLOW_CODE_ACTIONS`). The auto-generated behavior
   path no longer executes code at all (it appends a thought); any other
   `execute_python_code` action is rejected unless the env flag is set, and then sandboxed.

## 7b. Phase 4 — capability tags *(delivered 2026-06-10)*

### What changed

**`brain/data/capability_descriptions.json`** — extended from `{fn: "desc"}` into
the manifest `{fn: {"desc": "...", "tags": [...]}}` (88 entries, 84 tagged; the
old plain-string format remains accepted, so no migration cliff). Tag vocabulary:

| Tag(s) | Replaces |
|---|---|
| `outward`, `goal-progress` | `_USER_HELPFUL_FUNCTIONS` (union of both tags) |
| `introspective` | `_INTROSPECTION_FUNCTIONS` |
| `safe_to_explore` | `_SAFE_TO_EXPLORE` (Phase 2 ε-gate) |
| `procedural` | `step_execution._PROCEDURAL_FNS` (Executive-lane surface) |
| `outward_artifact` / `outward_explore` / `outward_sense` | the `_OUTWARD_HIGH/MED/LOW` standing-presence tiers |
| `mode_alert` / `mode_engaged` / `mode_wandering` / `mode_wandering_reflect` / `mode_drowsy` | the attention-mode inline fn tuples |
| `neuro_ne_focus` / `neuro_ne_suppress` / `neuro_calm_suppress` / `neuro_stress_suppress` / `neuro_stress_restore` | the neuromodulator boost lists |
| weighted `emo_<mode>:<w>` (e.g. `emo_focused:0.15`) | the `_emo_mode_map` per-fn weights |

**`select_function.py`** — `_load_manifest()` parses both formats into a desc map
+ a `{tag: {fn: weight}}` index; `_fns_tagged(*tags)`, `_tag_weights(tag)`, and
`_tagged_or(tags, default)` derive the sets. Every former name-list is now a
**tag-derived module set with its literal kept as fallback** — a missing/corrupt
manifest degrades to pre-Phase-4 behavior, never to an empty boost set. The
boost weights/multipliers stay in code (they are dynamics, not membership).

**`step_execution.py`** — `_PROCEDURAL_FNS` derives from the `procedural` tag
via a tiny local JSON read (no import of the heavy selector module → no cycle),
same fallback rule.

**E6 dead-entry cleanup (the §7 leftovers):** every residual
`pursue_committed_goal` data entry removed — `_SEMANTIC_PRIORS`
(confidence/motivation/expected_gain), `_EXECUTION_FNS`, `_USER_HELPFUL`,
the inline `_OUTWARD_MED`, `_emo_mode_map["focused"]`, and the neuro lists.
The stability-signal **persistence boost** (the one live-intent entry) was
relocated to `attend_goal`, the selectable "consciously stay with the goal"
proxy — the same relocation the commitment bias already received.

### Acceptance — met (`tests/brain/test_capability_tags.py`, 14 tests)

- **Golden:** every tag-derived set is asserted EQUAL to the literal name-list
  it replaced (and the emo-mode weights equal the old map) — same picks on any
  fixed context before/after, proven at the input level.
- **Participation:** a brand-new fn tagged `outward` in a probe manifest joins
  `_USER_HELPFUL_FUNCTIONS` with no code change.
- **Fallback:** a missing manifest yields the literal sets (never empty).
- **No dead entries:** `pursue_committed_goal` asserted absent from every data table.
- Full suite: **550 passed** (2 pre-existing unrelated embedder failures).

### Rollback

Delete the `tags` arrays from the manifest (or the file): `_tagged_or` falls
back to the literal sets and behavior is exactly pre-Phase-4. The dead-entry
removals are independent and safe to keep.
